## 1. 产品名称

**NeoCabin 3D 智舱车控 Demo**

一句话描述：通过可旋转、可点击、可语音控制的 3D 新能源车模型，演示智能座舱中直观且一致的车窗控制体验。

## 2. 项目背景和理解

### 2.1 项目背景

新能源汽车中控正从二维功能列表逐步转向以 3D 车模为核心的可视化控制。3D 车模既能表达车辆实时状态，也能让用户直接点击车身部件完成控制，减少在多层菜单中寻找功能的成本。

本项目需要用代码实现一个可独立演示的交互 Demo，验证以下核心能力：

1. 在桌面浏览器中呈现接近新能源车中控质感的 3D 车模。
2. 用户能够拖动车模，从不同角度观察车辆。
3. 用户能够直接点击车窗并看到对应的开关动画和状态反馈。
4. 用户能够通过中文语音命令控制单个或全部车窗。
5. 点击、语音和文本命令始终操作同一份车辆状态，避免界面与模型不同步。

### 2.2 产品目标

- 建立“看见车辆—直接操作—即时确认”的完整交互闭环。
- 验证 3D 部件点击与语音控车在同一界面中的协同体验。
- 形成可替换车模、可扩展控制部件的前端基础结构。
- 提供可本地运行、可现场演示、可按明确标准验收的 MVP。

### 2.3 目标用户

- 智能座舱产品、设计及研发团队：用于方案评审和交互验证。
- 客户、管理层及合作方：用于概念演示和能力展示。
- 前端及 3D 开发人员：用于后续扩展车门、灯光、后备箱等能力。

### 2.4 非目标

- 不连接真实车辆、CAN 总线或车控服务。
- 不实现账号、云端同步、数据持久化和多用户能力。
- 不承诺车规级实时性、安全性或功能安全要求。
- MVP 不包含车门、灯光、前后备箱、空调及车内视角控制。
- MVP 不进行移动端、小程序或原生 App 专项适配。

## 3. 核心 Idea

以 3D 车辆作为中控界面的主要交互对象，而不是装饰性模型。用户可通过三种入口控制车窗：

- **直接操作**：点击模型上的车窗，切换其开关状态。
- **自然语言**：说出“打开左前窗”等口令，系统识别并执行。
- **兼容降级**：在文本框输入同样的口令，模拟语音识别结果。

三种入口统一转换为标准车控命令，再由同一个车辆状态中心驱动 3D 动画、状态面板和操作反馈。这样既保证演示一致性，也便于后续扩展新的输入方式或可控部件。

## 4. 用户使用路径

### 4.1 首次进入

1. 用户通过 `localhost` 或 HTTPS 地址打开 Demo。
2. 页面显示加载进度和“正在加载 3D 车辆”提示。
3. 车模、材质和环境加载完成后进入默认待机状态。
4. 默认以左前 45° 视角展示整车，四个车窗初始状态均为关闭。
5. 页面以轻量提示告知用户“拖动旋转 · 点击车窗 · 语音控制”。

### 4.2 拖动观察

1. 用户在车模区域按下鼠标并拖动。
2. 车模跟随拖动方向旋转；垂直方向仅允许有限俯仰。
3. 松开鼠标后停止跟随，并保持当前观察角度。
4. 用户可点击“重置视角”回到默认视角。

### 4.3 点击控车

1. 用户将鼠标移动至可控车窗，车窗出现高亮或描边反馈，光标变为可点击状态。
2. 用户点击目标车窗。
3. 系统将点击转换为该车窗的 `toggle` 命令。
4. 车窗进入 `transitioning` 状态并播放升降动画。
5. 动画完成后，状态更新为 `open` 或 `closed`，状态面板和提示同步更新。

### 4.4 语音控车

1. 用户点击麦克风按钮。
2. 首次使用时，浏览器请求麦克风权限。
3. 授权成功后，界面进入“正在聆听”状态，用户说出标准中文口令。
4. 系统展示识别文本，并将有效文本解析为标准车控命令。
5. 命令执行后播放车窗动画，并显示“已打开左前窗”等结果反馈。
6. 对无效、含糊或无法识别的口令，不执行车控并提示支持的说法。

### 4.5 异常与降级路径

- 浏览器不支持语音识别：禁用麦克风按钮，展示原因和文本命令入口。
- 用户拒绝麦克风权限：展示权限提示，保留点击和文本命令能力。
- 语音识别失败或超时：结束聆听状态，不执行命令，允许立即重试。
- 3D 模型加载失败：展示错误占位、重试按钮和文本说明，不进入可控状态。
- 单个车窗动画执行中：忽略该车窗的重复命令，并提示“操作进行中”；其他车窗仍可操作。

## 6. MVP 范围

### 6.1 MVP 包含

- 单页桌面 Web Demo。
- 一辆许可清晰、可替换的 glTF/GLB 新能源车模型。
- 四个独立车窗：左前、右前、左后、右后。
- 四窗初始均关闭，仅支持全开和全关二态，不支持百分比开度。
- 鼠标拖动旋转、有限俯仰和重置视角。
- 模型车窗点击、悬停反馈及玻璃升降动画。
- 单窗及全部车窗的中文语音控制。
- 与语音解析规则一致的文本命令输入。
- 加载、空闲、操作中、成功、失败及兼容性提示。
- 本地开发和生产构建说明。

### 6.2 MVP 不包含

- 真实车控接口及真实车辆状态回传。
- 车窗半开、精确百分比或连续手势调节。
- 多轮语音对话、唤醒词、声纹、离线 ASR 或云端 ASR。
- 车门、车灯、天窗、后备箱、充电口及空调控制。
- 用户登录、埋点平台、数据库和后台管理系统。
- 手机、平板、小程序及原生 App 的交付适配。
- 特定汽车品牌或量产车型的 1:1 外观复刻。

## 7. 功能优先级

优先级采用以下定义：

- **P0**：缺失则无法完成核心演示闭环，必须在 MVP 中交付。
- **P1**：明显提升演示完整性，可在 P0 稳定后实现。
- **P2**：后续扩展方向，不影响 MVP 验收。

| 优先级 | 功能 |
| --- | --- |
| P0 | 3D 车模加载与展示、四窗节点识别、拖动旋转、点击单窗开关、车窗动画、统一状态管理、单窗/全窗语音命令、文本降级、基础操作反馈、加载失败处理 |
| P1 | 重置视角、车窗悬停描边、加载进度、语音识别原文展示、常见同义词、性能自适应、精细灯光与阴影 |
| P2 | 预设视角、车身颜色切换、昼夜环境、车门/车灯/前后备箱控制、移动端适配、云端语音服务、真实车辆接口 |

## 8. 功能需求与实现契约

### 8.1 建议技术方案

- 应用框架：React + TypeScript + Vite。
- 3D 渲染：Three.js + React Three Fiber，辅助能力可使用 Drei。
- 模型格式：glTF 2.0，生产环境优先使用单个 `.glb` 文件。
- 状态管理：React 状态或轻量状态库；必须只有一个车辆状态事实源。
- 语音识别：Chrome Web Speech API，识别语言设置为 `zh-CN`。
- 运行协议：开发环境使用 `localhost`，部署环境使用 HTTPS。

以上技术选型可在实现阶段等价替换，但不得改变本 PRD 定义的用户行为、状态契约和验收标准。

### 8.2 车辆状态模型

```ts
type WindowId = 'frontLeft' | 'frontRight' | 'rearLeft' | 'rearRight';
type WindowStableState = 'open' | 'closed';
type WindowState = WindowStableState | 'transitioning';

interface VehicleState {
  windows: Record<WindowId, WindowState>;
}
```

规则：

- 页面首次加载完成后，四个车窗均为 `closed`。
- 车窗动画开始前先进入 `transitioning`。
- 动画成功结束后，车窗更新为目标稳定状态。
- 动画中断时，车窗回退至动画开始前的稳定状态，并显示失败提示。
- 3D 模型、状态面板和操作反馈均读取同一份 `VehicleState`。

### 8.3 标准控车命令

```ts
type CommandSource = 'pointer' | 'voice' | 'text';
type CommandAction = 'open' | 'close' | 'toggle';
type CommandTarget = WindowId | 'allWindows';

interface VehicleCommand {
  source: CommandSource;
  target: CommandTarget;
  action: CommandAction;
}
```

- 点击车窗仅产生对应单窗的 `toggle` 命令。
- 语音和文本入口产生 `open` 或 `close` 命令。
- 全窗命令只操作不在目标状态中的车窗；已经达到目标状态的车窗保持不变。
- 同一车窗处于 `transitioning` 时，不接受新的点击、语音或文本命令。
- 全窗命令遇到正在动画的车窗时，跳过该车窗并继续操作其他可用车窗，同时反馈部分执行结果。
- 无法唯一确定目标或动作时，不生成命令、不改变车辆状态。

### 8.4 glTF/GLB 节点规范

模型至少需要提供以下可独立拾取和移动的车窗节点：

| 车窗 | 节点名 |
| --- | --- |
| 左前窗 | `window_front_left` |
| 右前窗 | `window_front_right` |
| 左后窗 | `window_rear_left` |
| 右后窗 | `window_rear_right` |

资产约束：

- 四个车窗必须为独立 Mesh 或具有独立父级节点，不能与车身合并为同一不可分割网格。
- 每个车窗的局部坐标轴和初始位置需要支持沿车门方向向下移动。
- 玻璃材质应使用透明或半透明 PBR 材质，并保留清晰的可见边界。
- 模型替换时必须维持上述节点名，或在独立配置映射中提供等价节点关系。
- 车模资产必须附来源、许可证、作者及允许使用范围说明；无法确认授权的资产不得进入最终交付。

### 8.5 语音和文本命令规则

#### 标准命令

| 目标 | 打开命令示例 | 关闭命令示例 |
| --- | --- | --- |
| 左前窗 | 打开左前窗 | 关闭左前窗 |
| 右前窗 | 打开右前窗 | 关闭右前窗 |
| 左后窗 | 打开左后窗 | 关闭左后窗 |
| 右后窗 | 打开右后窗 | 关闭右后窗 |
| 全部车窗 | 打开全部车窗 | 关闭全部车窗 |

#### 可接受同义表达

- 动作：“打开”“开启”“降下”映射为 `open`；“关闭”“关上”“升起”映射为 `close`。
- 目标：“左前车窗”“左前窗”“驾驶位车窗”映射为 `frontLeft`。
- 目标：“右前车窗”“右前窗”“副驾驶车窗”映射为 `frontRight`。
- 目标：“左后车窗”“左后窗”映射为 `rearLeft`。
- 目标：“右后车窗”“右后窗”映射为 `rearRight`。
- 目标：“全部车窗”“所有车窗”“四个车窗”映射为 `allWindows`。

解析前允许忽略空格、常见中文标点及“请”“帮我”“把”等礼貌或连接词。MVP 不处理“开一点”“开一半”“除了左前窗都关闭”等复杂表达。

#### 无效和含糊命令

- 只有动作没有目标，如“打开车窗”：不执行，并提示用户指定车窗或说“全部车窗”。
- 只有目标没有动作，如“左前窗”：不执行，并提示补充打开或关闭动作。
- 同时包含冲突动作，如“打开再关闭左前窗”：不执行，并提示重新表达。
- 与车控无关或无法识别：不执行，并展示可用命令示例。

## 9. 关键交互说明

### 9.1 页面布局

- 顶部：产品名称、轻量运行状态和必要的帮助入口。
- 中央主区域：3D 车模，占据主要视觉空间，默认完整展示车辆。
- 右侧或右下控制区：四窗状态、麦克风按钮、识别文本和文本命令入口。
- 底部轻提示：拖动旋转、点击车窗、语音控制的使用说明。
- 重置视角按钮不得遮挡车辆主要点击区域。

### 9.2 视角与拖动

- 鼠标左键拖动控制环绕视角，拖动方向与用户对物体旋转的直觉一致。
- 水平允许连续环绕；垂直俯仰角建议限制在 15° 至 75°，禁止翻转到地面以下。
- 禁止相机进入车体内部或远离到无法识别车窗的位置。
- 拖动位移超过 5 px 时判定为旋转操作，松开时不得误触车窗点击。
- MVP 不要求惯性旋转和滚轮缩放；若实现滚轮缩放，必须设置安全的远近边界。

### 9.3 车窗点击与动画

- 悬停可控车窗时使用青蓝色描边、轻微提亮或透明罩层进行反馈，不改变车窗实际状态。
- 点击命中区域应覆盖完整玻璃表面，不使用与模型位置脱离的固定二维热区。
- 关闭到打开：玻璃沿车门内部方向向下移动至预设终点。
- 打开到关闭：玻璃返回模型初始位置。
- 单次动画建议为 600–900 ms，必须在 1 秒内完成。
- 动画使用缓入缓出，不出现瞬移、明显穿出车门或越过终点。
- 动画期间该车窗保留操作中反馈，且不响应重复命令。

### 9.4 语音交互状态

| 状态 | 界面表现 | 允许操作 |
| --- | --- | --- |
| 未启动 | 显示麦克风按钮和示例口令 | 可开始语音、点击或输入文本 |
| 请求权限 | 显示“等待麦克风授权” | 可取消，不触发车控 |
| 聆听中 | 麦克风高亮或脉冲，显示“正在聆听” | 可主动停止；点击控车仍可用 |
| 识别成功 | 展示识别原文和解析结果 | 自动执行有效命令 |
| 无效口令 | 展示原因及示例，不执行命令 | 可立即重试或使用文本入口 |
| 识别失败/超时 | 结束动效并显示重试提示 | 可立即重试 |
| 不支持/无权限 | 麦克风禁用并说明原因 | 点击和文本入口保持可用 |

语音识别采用单次命令模式：点击后监听一句口令，获得最终识别结果、发生错误或超时后自动结束，不持续监听。

### 9.5 状态反馈

- 成功反馈必须包含目标和结果，例如“左前窗已打开”。
- 当前状态与命令目标相同时，不重复播放动画，提示“左前窗已经打开”。
- 失败反馈必须说明未执行原因，不能只显示通用“出错了”。
- 临时提示建议停留 2–3 秒；麦克风权限、语音不支持和模型加载失败等持续性问题不得自动消失。

## 10. 视觉风格要求

### 10.1 整体风格

- 参考新能源智能座舱中控：科技、克制、安静、精密，不使用游戏化或高饱和霓虹风格。
- 主背景采用深灰至近黑渐变，以车辆为视觉中心。
- 反馈色采用克制的青蓝色；成功、警告和错误可分别使用低饱和绿、橙、红。
- 控制面板采用轻量半透明深色面板或无边界布局，避免大面积遮挡车模。

### 10.2 3D 表现

- 车漆具有清晰但不过曝的环境反射，车窗具备合理透明度和高光。
- 使用柔和棚拍式主光、轮廓光和环境光，保证深色车身边缘可辨识。
- 地面保留柔和接触阴影，车辆不能呈现悬浮感。
- 默认视角下车辆主体完整可见，四个车窗可通过旋转观察。
- 加载期间使用简洁进度反馈，不展示未完成材质的闪烁模型。

### 10.3 UI 与动效

- 字体优先使用系统中文无衬线字体，正文需在目标分辨率下清晰可读。
- 主要按钮可点击区域不小于 40 × 40 px。
- 状态变化动效以 150–300 ms 为主，车窗机械动画以 600–900 ms 为主。
- 动效用于表达状态变化，不使用持续干扰用户观察的装饰动画。
- 在 1280 × 720 至 1920 × 1080 范围内，车模、语音入口和状态信息不得互相遮挡或溢出。

## 11. 验收标准

### 11.1 功能验收

| 编号 | 验收项 | 通过标准 |
| --- | --- | --- |
| AC-01 | 页面启动 | 在最新版桌面 Chrome 通过 `localhost` 或 HTTPS 正常打开，无阻断性控制台错误 |
| AC-02 | 初始状态 | 模型加载完成后车辆完整可见，四窗状态均为关闭 |
| AC-03 | 拖动旋转 | 可连续水平环绕并有限俯仰，松手后保持视角，不翻转、不进入地面以下 |
| AC-04 | 拖动防误触 | 明显拖动后松开鼠标不触发车窗开关 |
| AC-05 | 重置视角 | 点击后恢复默认左前 45° 视角，车辆完整处于画面内 |
| AC-06 | 车窗命中 | 四个车窗均可独立悬停和点击，点击其他车身区域不改变车窗状态 |
| AC-07 | 单窗动画 | 点击任一车窗后，在 1 秒内完成方向正确、可辨识的升降动画 |
| AC-08 | 点击状态同步 | 动画结束后，3D 位置与状态面板中的开关状态一致 |
| AC-09 | 单窗语音 | 每个标准单窗口令均能控制正确车窗，不影响其他车窗 |
| AC-10 | 全窗语音 | “打开全部车窗”和“关闭全部车窗”可使所有可用车窗达到目标状态 |
| AC-11 | 文本降级 | 文本入口支持全部标准口令，执行结果与语音入口一致 |
| AC-12 | 统一状态 | 点击、语音和文本交替操作后，模型、状态面板和结果提示始终一致 |
| AC-13 | 重复命令 | 动画中的同一车窗不重复执行；已达到目标状态时不重复播放动画 |
| AC-14 | 无效口令 | 缺少目标、缺少动作、动作冲突和无关口令均不改变车辆状态，并给出可操作提示 |
| AC-15 | 语音异常 | 不支持语音、权限拒绝、识别失败和超时均有明确反馈，点击和文本控制仍可用 |
| AC-16 | 模型失败 | 模型加载失败时显示原因和重试入口，不出现可点击但无响应的伪可用状态 |

### 11.2 性能与兼容验收

- 在项目指定的主流开发机和稳定网络环境下，首次进入后 5 秒内达到可旋转、可点击状态。
- 稳定交互阶段目标帧率不低于 45 FPS；车窗动画期间不出现肉眼明显的长时间卡顿。
- 1280 × 720、1440 × 900、1920 × 1080 三种视口下完成布局检查，无核心信息遮挡、裁切或横向滚动条。
- 模型与纹理应经过 Web 优化；建议首屏 3D 资源总量不超过 15 MB。若超出，需要提供压缩方案或说明原因。
- 语音能力仅以最新版桌面 Chrome 为正式验收环境；其他浏览器只验收点击和文本降级能力。

### 11.3 验收测试场景

至少覆盖以下组合场景并保留结果：

1. 依次点击打开四个车窗，再逆序关闭。
2. 快速重复点击同一车窗。
3. 一个车窗动画中控制另一个车窗。
4. 点击打开后使用语音关闭，再使用文本打开。
5. 部分车窗已打开时执行“打开全部车窗”。
6. 输入或说出无效、含糊和冲突口令。
7. 首次拒绝麦克风权限后继续使用文本控制。
8. 模拟浏览器不支持 Web Speech API。
9. 模拟模型地址错误并执行重试。
10. 在三种目标分辨率下旋转至车辆四个方向并操作可见车窗。

## 12. 风险和降级方案

| 风险 | 影响 | 预防措施 | 降级方案 |
| --- | --- | --- | --- |
| 模型版权或使用范围不明确 | 无法合法交付或展示 | 使用许可清晰的资产并随项目记录来源和许可证 | 替换为自制或允许商用的通用模型 |
| 模型未拆分车窗节点 | 无法拾取或制作独立动画 | 接入前按节点规范验模 | 由 3D 工具拆分玻璃；仍不可用时采用独立玻璃替代 Mesh |
| 模型面数或纹理过大 | 加载慢、帧率低 | 压缩纹理、减少材质、合并非交互网格、使用 Draco/KTX2 | 降低阴影和像素比，使用低模资产 |
| Web Speech API 兼容性有限 | 无法使用真实语音输入 | 明确最新版 Chrome、localhost/HTTPS 环境 | 禁用麦克风并启用文本命令入口 |
| 麦克风权限被拒绝 | 语音路径中断 | 首次请求前说明用途 | 保留点击和文本控制，提示如何重新授权 |
| 语音误识别或口令含糊 | 错控目标车窗 | 使用有限语法、展示识别原文、含糊时不执行 | 提示标准口令，允许文本输入 |
| 车窗动画穿模 | 降低写实感 | 为每个车窗配置独立终点并进行四向视角检查 | 缩短移动距离或改用透明度渐隐作为临时降级 |
| 并发命令导致状态不一致 | 模型和 UI 冲突 | 使用统一状态机并锁定动画中的单个车窗 | 忽略冲突命令，回退到上一个稳定状态 |
| 模型资源加载失败 | 核心界面不可用 | 本地托管、校验资源路径、提供错误捕获 | 展示错误占位和重试入口，不伪造可用状态 |
| 演示现场网络不稳定 | 外部资源或语音服务不可用 | 所有静态资产随项目本地打包 | 点击和文本能力离线可用；语音不可用时明确提示 |
