## 3. 实现了哪些关键逻辑

- **窗口配置**：集中管理 `WindowId → nodeName` 映射，前窗偏移 -0.35、后窗偏移 -0.3（Y 轴向下），动画时长 750ms。配置独立于业务逻辑，模型节点名变化只需修改此文件。
- **窗口注册**：`useWindowInteraction` 在 useEffect 中通过 `sceneRoot.getObjectByName(config.nodeName)` 发现四个窗口节点，递归 `findFirstMesh` 查找第一个 Mesh 子孙，保存初始 `position.y`。节点缺失或 Mesh 不存在时 throw（触发 ErrorBoundary）。
- **悬停反馈**：`handlePointerEnter/Leave` 遍历 hit object 祖先查找 WindowId；匹配时设置 `material.emissive` 为青蓝色 + `emissiveIntensity=0.45`，离开时清零 emissive。同步设置 `document.body.style.cursor`。
- **点击→命令**：`handleClick` 先查 `findWindowId`（遍历 Object3D.parent 链），再查 `isDragRef.current`（NC-006 drag guard），再查 `useVehicleStore.getState().windows[id] === 'transitioning'`（动画锁）。通过后调用 `executeCommand({ source: 'pointer', target: id, action: 'toggle' })`。若返回 accepted/partial，记录动画起始/目标 Y 坐标。
- **动画驱动**：`useFrame` 每秒迭代 `animatingRef` 中所有活跃动画。每帧累加 `delta * 1000` ms，计算 `progress = elapsed / ANIMATION_DURATION_MS`，应用 `easeInOutCubic`。mesh.position.y 从 startY 插值到 targetY。progress ≥ 1 时 snap 到精确终值，调用 `completeWindowTransition(windowId)`。
- **失败回退**：useEffect cleanup 遍历活跃动画调用 `failWindowTransition` 回退到前一稳定状态；clearHover 清理残留高亮。
- **与状态机的关系**：3D 动画只读 vehicleStore 状态判断是否 transitioning，只写 mesh.position.y。状态变更（transitioning→open/closed）由 store action 完成。车窗动画期间重复点击被 store.executeCommand 的 transitioning 检查拦截（返回 blocked）。

### Sub-agent 使用情况

- 委派内容：本任务由主代理直接实现，未委派 Sub-agent。原因：useWindowInteraction 与 VehicleModel 紧密耦合（事件 handler 需挂载到 `<primitive>`），拆分会增加集成成本。
- 主代理审查：自检窗口注册异常路径、动画插值数学、状态机同步、drag guard 时序、hover/click handler 闭包正确性和测试覆盖。
