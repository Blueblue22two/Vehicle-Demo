# 分阶段任务执行工作流

## 1. 目的与适用范围

本文规范 AI 按照 `memory_bank/task-list.md` 的顺序实现 NeoCabin 项目任务时必须遵循的标准流程。每次只处理一个任务；当前任务验收完成并更新状态后，才能进入下一任务。

`task-list.md` 是任务顺序、依赖和状态的唯一事实来源；`docs/3d-vehicle-control-demo-prd.md` 定义产品边界；`memory_bank/technology-stack.md` 定义技术约束；`memory_bank/report_template.md` 定义任务报告格式。

## 2. 角色与职责

- **主代理**：读取上下文、确认任务、制定计划、拆分工作、委派 Sub-agent、审查代码、运行最终验证、修复集成问题、更新文档并输出报告。主代理对最终结果负责。
- **编码 Sub-agent**：完成边界清晰的代码子任务和对应测试，不得自行改变产品范围、公共契约、任务优先级或其他任务状态。
- **修复 Sub-agent**：根据失败日志定位并修复限定范围内的 Bug；不得通过删除测试、弱化断言或跳过质量门禁来制造通过结果。

主代理应避免多个代理同时修改同一文件。委派前必须说明任务 ID、允许修改的文件/模块、验收条件和禁止事项；Sub-agent 完成后，主代理必须独立审查和验证。

## 3. 任务选择规则

1. 从 `task-list.md` 总览中找到顺序最前且状态为 `TODO` 的任务。
2. 确认该任务所有依赖均为 `DONE`；依赖未完成时不得跳过顺序实施。
3. 若已有 `DOING` 任务，优先继续该任务，不得并行开启另一个阶段任务。
4. P0 全部完成后才能开始 P1；P2 不属于 MVP 完成条件。
5. 开始实现时，将任务总览和任务详情中的状态同步更新为 `DOING`。
6. 若存在无法在当前范围解决的外部阻塞，将状态更新为 `BLOCKED`，并在阶段报告记录证据和解除条件。

## 4. 单任务标准工作流

### 步骤 1：读取相关文档

至少读取：

- `AGENTS.md`
- `docs/3d-vehicle-control-demo-prd.md`
- `memory_bank/technology-stack.md`
- `memory_bank/task-list.md`
- `memory_bank/stage_workflow.md`
- `memory_bank/report_template.md`

同时检查当前工作区状态、相关源码、测试和最近任务报告。不得覆盖与当前任务无关的用户修改。

### 步骤 2：确认当前任务

输出并核对以下信息：任务 ID、名称、优先级、依赖、当前状态、允许范围、验收标准、验收命令、风险点。若任务描述与 PRD 冲突，以 PRD 为产品基准并停止实现，先报告冲突。

### 步骤 3：输出当前阶段计划

计划必须包含：

1. 将修改的模块和预期行为。
2. Sub-agent 的具体分工和文件边界。
3. 先写或补充哪些测试。
4. 需要执行的验证命令。
5. 风险、回退方式和完成定义。

计划应仅覆盖当前任务，不提前实现后续任务。

### 步骤 4：按计划实现代码

1. 主代理将边界清晰的编码工作委派给 Sub-agent。
2. Sub-agent 按现有架构实现代码和测试，并返回修改摘要、风险与已运行命令。
3. 主代理审查全部差异，检查范围、类型契约、状态一致性、错误处理和测试质量。
4. 主代理完成必要的集成调整；不得未经审查直接接受 Sub-agent 输出。
5. 若当前任务不适合拆分，主代理仍需明确记录原因，再由单一代理实现，避免为了委派而制造冲突。

### 步骤 5：运行验证命令

先执行当前任务详情中的“验收命令”，再执行受影响范围的静态检查。至少保存命令、退出码和关键输出；不得只写“验证通过”。

若任务要求的 script 尚不存在，只有当创建该 script 属于当前任务范围时才能补充，否则将任务标记为 `BLOCKED` 并报告缺失依赖。

### 步骤 6：运行测试案例

逐条执行当前任务列出的测试案例，并记录 `PASS`、`FAIL` 或 `BLOCKED`。自动化不能可靠覆盖的真实麦克风、3D 视觉或性能检查，必须执行人工验收并记录环境、步骤和证据，不能伪造为自动化通过。

### 步骤 7：修复 Bug 并回归

1. 根据失败日志定位根因，禁止通过跳过测试、放宽断言或删除功能规避问题。
2. 可将边界明确的修复委派给修复 Sub-agent，并提供失败命令、日志、预期行为和修改范围。
3. 每次修复后重跑失败用例，再重跑当前任务全部验收命令。
4. 连续三轮仍无法解决时停止扩改，将任务标记为 `BLOCKED`，记录已尝试方案和所需外部输入。

### 步骤 8：输出并保存变更报告

严格使用 `memory_bank/report_template.md`。报告保存到：

```text
memory_bank/reports/<TASK-ID>.md
```

同一任务重新执行时更新原报告，并保留新的验证结果。报告中的命令结果、测试结果和遗留问题必须真实、可复核。

### 步骤 9：更新任务状态和信息

仅当代码、测试、验收命令和报告全部完成后，才能把任务状态更新为 `DONE`。同步更新：

- `task-list.md` 总览表中的状态。
- 对应任务详情中的 `TODO 状态`。
- 必要时在任务详情补充实际实现偏差、验收证据或已接受风险，但不得静默改写原验收标准。

若未完成，则使用 `DOING` 或 `BLOCKED`，并在报告中写明原因。状态更新后再次检查两个位置一致。

## 5. 完成门禁

一个任务被视为完成，必须同时满足：

- 实现未超出当前任务和 PRD 范围。
- Sub-agent 产出已由主代理审查。
- 当前任务要求的自动化和人工测试均已执行。
- 所有验收命令成功，或已记录并获接受的非阻断例外。
- 没有通过禁用测试、忽略错误或提交无授权资产绕过门禁。
- 阶段报告已保存且内容完整。
- `task-list.md` 总览与详情状态均为 `DONE`。

## 6. 最终输出格式

主代理在聊天中简要说明：

1. 已完成的任务 ID 与结果。
2. 关键变更和验证结论。
3. 阶段报告文件路径。
4. 遗留问题或下一任务 ID。

详细内容以落盘的阶段报告为准，聊天输出不得替代报告。
