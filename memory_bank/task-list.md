## 1. 执行规则

- 按任务编号顺序执行；只有“依赖”全部为 `DONE` 后才能开始当前任务。
- P0 全部完成并通过验收后，才进入 P1；P2 不属于 MVP 交付门槛。
- 每个任务只有在代码、测试、验收命令和人工检查均通过后才能标记为 `DONE`。
- 若任务因模型授权、节点缺失或浏览器能力被阻塞，标记为 `BLOCKED` 并记录原因，不得以临时伪实现冒充完成。
- 文档中的验收命令以项目根目录为执行位置；命令所需 script 应在对应基础任务中建立。

  3. 建立 `app/domain/features/scene/components/test/types` 目录边界。
  4. 配置 TypeScript strict、ESLint、Prettier、Vitest、Testing Library 和 Playwright。
  5. 建立 `dev/build/typecheck/lint/format:check/test/test:e2e/validate:model/check` scripts。
- **TODO 状态**：`DONE`
- **测试案例**：
  - 全新安装后开发服务器可启动。
  - 示例单元测试可运行。
  - 空白应用可生产构建并由 preview 服务打开。
- **验收标准**：`npm ci` 可复现安装；类型、Lint、格式、测试及构建脚本均存在且返回成功；目录结构符合技术栈文档。

### NC-002 建立车辆领域类型与统一状态机

- **任务描述**：实现 `WindowId`、`WindowState`、`VehicleCommand`、车辆 store 和命令执行器，作为所有入口的唯一状态源。
- **任务优先级**：P0
- **依赖**：NC-001
- **任务实现思路**：
  1. 按 PRD 定义领域类型，初始四窗均为 `closed`。
  2. 使用 Zustand 建立车辆状态和反馈状态，不在 store 中保存 Three.js 对象。
  3. 实现 `open/close/toggle`、单窗和 `allWindows` 命令。
  4. 实现幂等判断、单窗动画锁、上一个稳定状态和成功/失败收尾 action。
  5. 全窗命令跳过 `transitioning` 车窗，并返回完整或部分执行结果。
- **TODO 状态**：`DONE`
- **测试案例**：
  - 初始四窗均关闭。
  - 单窗打开、关闭、切换产生正确目标状态。
  - 已打开车窗再次打开不进入动画。
  - 动画中同窗新命令被拒绝，其他车窗可执行。
  - 全窗命令跳过锁定车窗并操作其他车窗。
  - 动画失败回退至上一个稳定状态。
- **验收标准**：所有输入方式可复用同一 `executeCommand`；状态机不存在无法收敛的 `transitioning` 分支；单元测试覆盖正常、幂等、冲突和回退。

### NC-003 建立中文车控命令解析器

- **任务描述**：将语音或文本中的中文口令解析为标准 `VehicleCommand`，并为无效、含糊和冲突表达返回稳定错误类型。
- **任务优先级**：P0
- **依赖**：NC-001、NC-002
- **任务实现思路**：
  1. 建立纯函数 `normalizeCommandText` 和 `parseVehicleCommand`。
  2. 清理空格、标点及“请、帮我、把”等非语义词。
  3. 使用显式词典映射打开/关闭动作、四窗目标和全窗目标。
  4. 检测缺少动作、缺少目标、动作冲突、多目标歧义和不支持表达。
  5. 解析器接收 `voice | text` 来源并生成带正确 source 的命令。
- **TODO 状态**：`DONE`
- **测试案例**：
  - 10 条标准单窗命令和全窗命令正确解析。
  - “开启、降下、关上、升起”等同义词正确映射。
  - “驾驶位车窗”“副驾驶车窗”正确映射。
  - 礼貌词、空格和常见标点不影响结果。
  - “打开车窗”“左前窗”“打开再关闭左前窗”“开一半”均不生成命令。
- **验收标准**：PRD 中全部标准命令和同义表达均通过；无效命令不改变车辆状态；解析器不依赖 DOM 或语音 API。
- **验收命令**：
  ```bash
  npm run test -- src/features/command
  npm run typecheck
  ```
- **可能的风险点**：简单字符串包含关系造成动作误判；同义词互相覆盖；过度支持自然语言导致不可预测行为。
